# Toss 결제 시나리오 정리

기준일: 2026-03-10

이 문서는 토스페이먼츠 공식 개발자 문서를 기준으로, 현재 프로젝트에서 참고해야 할 결제 시나리오를 운영 관점에서 재구성한 문서다.

## 1. 전체 시나리오 분류

토스페이먼츠가 제공하는 주요 결제 시나리오는 아래처럼 나눌 수 있다.

1. 결제위젯 연동
2. 결제창 연동
3. API 직접 연동
4. 브랜드페이 연동
5. 자동결제(빌링) 연동
6. 웹훅 기반 비동기 처리
7. 결제 취소/조회 운영 시나리오
8. 지급대행(Payout) 시나리오

---

## 2. 결제위젯 시나리오

가장 표준적인 온라인 결제 시나리오다.

흐름:

1. 상점 클라이언트가 결제위젯 UI를 렌더링한다.
2. 구매자가 결제수단을 선택한다.
3. 토스 결제창 또는 연계 앱으로 이동한다.
4. 결제 성공 시 `successUrl`로 리다이렉트된다.
5. 서버가 `paymentKey`, `orderId`, `amount`로 승인 API를 호출한다.
6. 승인 성공 후 주문/결제 상태를 확정한다.

특징:

- 결제 UI를 직접 만드는 부담이 가장 적다.
- 카드, 계좌이체, 가상계좌, 간편결제 등 다수 수단을 한 번에 붙이기 쉽다.
- 약관 UI도 같이 제공된다.
- 운영자가 일부 UI 구성을 관리자 화면에서 바꿀 수 있다.

적합한 경우:

- 웹 체크아웃을 빠르게 붙여야 할 때
- 결제수단을 자주 바꿔야 할 때
- 자체 결제 UI보다 안정성과 속도가 중요할 때

---

## 3. 결제창 시나리오

위젯보다 UI 자유도는 낮지만 구조가 단순한 방식이다.

흐름:

1. 클라이언트에서 `requestPayment()`로 결제창을 연다.
2. 구매자가 카드/간편결제/계좌이체 등을 진행한다.
3. 성공 시 `successUrl`, 실패 시 `failUrl`로 이동한다.
4. 서버가 승인 API를 호출한다.

특징:

- 토스가 제공하는 결제창을 그대로 사용한다.
- 웹/모바일 웹 기준으로 붙이기 쉽다.
- 우리 프로젝트처럼 앱 안에서 WebView로 결제창을 여는 패턴과 잘 맞는다.

적합한 경우:

- 체크아웃 UI는 직접 유지하되 결제창은 토스 표준 UI를 쓰고 싶을 때
- 앱/WebView 구조에서 빠르게 붙일 때

---

## 4. API 직접 연동 시나리오

상점이 결제 진입 UX를 더 많이 통제하는 방식이다.

흐름:

1. 서버가 `POST /v1/payments`로 결제 객체를 만든다.
2. 응답의 `checkout.url` 또는 지정된 결제창 정보를 사용해 구매자를 결제 플로우로 보낸다.
3. 결제 완료 후 서버가 승인 또는 후속 처리 API를 호출한다.
4. 웹훅으로 비동기 상태를 보완한다.

특징:

- Hosted/Direct 흐름을 직접 제어할 수 있다.
- 자체 결제 화면과의 결합도가 높다.
- 대신 개발량과 운영 복잡도가 커진다.

적합한 경우:

- 결제 진입 흐름을 서버 중심으로 통제해야 할 때
- 다중 결제수단 제어를 세밀하게 해야 할 때
- 웹훅과 재시도, 대사 로직까지 직접 설계할 준비가 되어 있을 때

주의:

- 토스 공식 가이드도 서버-서버 방식에서는 웹훅 사용을 강하게 권장한다.

---

## 5. 브랜드페이 시나리오

상점 전용 지갑/간편결제 UX를 붙이는 시나리오다.

흐름:

1. 상점이 브랜드페이 사용자를 인증한다.
2. 저장된 결제수단을 조회한다.
3. 브랜드페이 결제를 승인한다.
4. 필요 시 자동결제나 결제수단 관리 API를 같이 쓴다.

특징:

- 재구매 UX가 빠르다.
- 카드/계좌 등 저장 결제수단을 재사용할 수 있다.
- 회원/약관/결제수단 상태를 따로 관리해야 한다.

적합한 경우:

- 재결제가 잦은 서비스
- 결제 이탈률을 줄이는 것이 핵심인 서비스
- 자체 회원 체계와 결제수단 저장 정책이 명확한 서비스

---

## 6. 자동결제(빌링) 시나리오

정기결제나 재청구에 적합한 시나리오다.

흐름:

1. 최초 1회 결제수단 등록 또는 인증을 받는다.
2. `authKey` 기반으로 빌링키를 발급한다.
3. 이후 서버가 빌링키로 자동결제를 승인한다.
4. 실패 시 재시도/알림/수금 실패 처리 로직을 둔다.

적합한 경우:

- 구독형 상품
- 월별 수수료 청구
- 주기적 자동 청구

주의:

- 최초 동의/인증 흐름과 이후 자동청구 흐름을 분리해서 설계해야 한다.

---

## 7. 비동기 결제 시나리오

동기 승인만으로 끝나지 않는 결제수단에 필요한 시나리오다.

대표 사례:

- 가상계좌
- 일부 계좌이체/간편결제 후속 상태 반영
- 취소/환불 후속 반영
- 지급대행 상태 변경

흐름:

1. 상점이 결제를 요청한다.
2. 즉시 승인되지 않거나 후속 상태 변경이 발생한다.
3. 토스가 웹훅을 보낸다.
4. 서버가 이벤트를 멱등 처리한다.
5. 주문, 결제, 정산 상태를 보정한다.

웹훅 주요 이벤트:

- `PAYMENT_STATUS_CHANGED`
- `DEPOSIT_CALLBACK`
- `CANCEL_STATUS_CHANGED`
- `payout.changed`
- `seller.changed`

운영 원칙:

- 웹훅은 중복 수신을 전제로 멱등 처리해야 한다.
- 웹훅만 믿지 말고 조회/재시도 배치도 둬야 한다.
- 토스는 웹훅 실패 시 재전송한다.

---

## 8. 승인/조회/취소 시나리오

결제 운영에 반드시 필요한 기본 시나리오다.

핵심 API:

- 승인: `POST /v1/payments/confirm`
- 결제 조회: `GET /v1/payments/{paymentKey}`
- 주문번호 조회: `GET /v1/payments/orders/{orderId}`
- 결제 취소: `POST /v1/payments/{paymentKey}/cancel`

운영 시나리오:

1. 성공 리다이렉트 직후 승인
2. 승인 실패 시 재시도 또는 고객 안내
3. 승인 후 상태 불일치 시 조회 API로 대사
4. 환불/취소 발생 시 내부 정산 상태 동기화

주의:

- 승인 API는 인증 완료 후 유효 시간 안에 호출해야 한다.
- `paymentKey`, `orderId`, `amount`는 서버에서 검증해야 한다.
- `orderId`는 충분히 랜덤하고 고유해야 한다.

---

## 9. 지급대행(Payout) 시나리오

오픈마켓이나 중개형 서비스에서 판매자/기사에게 돈을 보내는 시나리오다.

흐름:

1. 셀러를 등록하거나 상태를 조회한다.
2. 지급 가능 잔액을 확인한다.
3. `POST /v2/payouts`로 지급 요청을 보낸다.
4. `payout.changed`, `seller.changed` 웹훅으로 후속 상태를 받는다.

특징:

- 일반 결제와 다르게 보안 키 기반 암호화(JWE)가 필요하다.
- 셀러 상태와 지급 가능 상태를 먼저 확인해야 한다.
- 테스트/운영 전환 시 실송금 영향이 크므로 검증이 더 엄격해야 한다.

적합한 경우:

- 기사 정산
- 판매자 정산
- 플랫폼 수익 분배

---

## 10. Barotruck 기준 추천 시나리오

현재 프로젝트 구조를 기준으로 보면 우선순위는 아래가 맞다.

1. 화주 결제
- `결제창 or 위젯 + 서버 승인 + 웹훅`
- 현재 코드 구조상 `prepare -> 결제창 -> confirm` 흐름을 유지해도 된다.

2. 차주 정산/지급
- `payout API + seller 상태 관리 + payout webhook`
- 지급 성공/실패를 정산 상태와 분리해서 관리해야 한다.

3. 운영 보강
- 결제 조회 API 대사
- 웹훅 멱등 처리
- 재시도 큐
- 취소/환불 후 상태 동기화

즉, 이 프로젝트에서 실무적으로 필요한 최소 시나리오는 아래 4개다.

1. 결제 요청
2. 결제 승인
3. 웹훅 후속 반영
4. 기사 지급

---

## 11. 현재 프로젝트와 매핑

현재 백엔드 기준으로 이미 들어가 있거나 맞물리는 기능은 아래다.

- 결제 준비/승인: `TossPaymentService`
- 결제 승인 HTTP 호출: `TossPaymentHttpClient`
- 웹훅 수신: `/api/v1/payments/webhooks/toss`
- 차주 지급: `TossDriverPayoutGatewayClient`

실제 운영 기준 필수 환경변수:

- `TOSS_CLIENT_KEY`
- `TOSS_SECRET_KEY`
- `TOSS_WEBHOOK_SECRET`
- `TOSS_PAYOUT_SECRET_KEY`
- `TOSS_PAYOUT_SECURITY_KEY`
