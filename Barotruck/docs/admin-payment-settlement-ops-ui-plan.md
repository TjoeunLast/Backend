# 관리자 결제/정산 운영 UI 기획안

## 1. 목적

관리자 페이지에서 결제와 정산 과정을 더 자유롭고 안전하게 제어할 수 있도록 운영 중심 UX/UI 구조를 정의한다.

이 문서는 다음 목표를 가진다.

- 상태를 한눈에 분리해서 볼 수 있게 한다.
- 운영자가 다음 액션을 즉시 판단할 수 있게 한다.
- 내부 상태와 PG 상태 불일치를 빠르게 발견하고 조치하게 한다.
- 지급 실패, 분쟁, 보류 같은 예외 처리를 별도 운영 흐름으로 분리한다.
- 대량 처리와 감사 로그를 고려한 운영 콘솔 구조를 제안한다.

---

## 2. 핵심 원칙

### 2.1 상태를 절대 섞지 않는다

관리 화면에서 아래 4개 상태는 반드시 분리해서 보여줘야 한다.

1. 운송 상태
2. 결제 상태
3. 정산 상태
4. 지급 상태

예시:

- 운송 상태: `REQUESTED`, `ACCEPTED`, `LOADING`, `IN_TRANSIT`, `UNLOADING`, `COMPLETED`
- 결제 상태: `READY`, `PAID`, `CONFIRMED`, `CANCELLED`, `REFUNDED`, `DISPUTED`
- 정산 상태: `WAIT`, `READY`, `HOLD`, `COMPLETED`
- 지급 상태: `REQUESTED`, `PROCESSING`, `SUCCESS`, `FAILED`

운영자가 헷갈리는 가장 큰 원인은 서로 다른 도메인 상태를 한 카드나 한 배지에 섞어 보여주는 것이다. 이 문서에서 제안하는 모든 UI는 이 4분리를 전제로 한다.

### 2.2 조회 화면이 아니라 운영 콘솔이어야 한다

단순 목록 조회가 아니라 다음 액션이 바로 보이고 실행 가능한 운영 콘솔 형태가 되어야 한다.

### 2.3 모든 강제 처리에는 근거가 남아야 한다

강제 확인, 강제 완료, 보류, 재지급, 환불 같은 조치는 반드시 아래를 남겨야 한다.

- 변경자
- 변경 시각
- 변경 전 상태
- 변경 후 상태
- 사유

4. 지급 요청됨
5. 지급 실패
6. 분쟁/보류

각 큐 카드에는 다음을 표시한다.

- 건수
- 총 금액
- 최근 1시간 변동

## 6. 상세 운영 페이지 UX 제안

상세 페이지는 `상태 비교 패널 + 운영 액션 패널` 2축으로 설계한다.

추천 타임라인 항목:

1. 주문 생성
2. 배차 확정
3. 운송 완료
4. 결제 승인
5. 차주 확인
6. 정산 READY 전환
7. 지급 요청
8. 지급 완료
9. 환불/취소

### 6.2 우측: 운영 액션 패널

추천 액션:

1. PG 실조회
2. 내부 상태 동기화
3. 차주 확인 강제 처리
4. 정산 HOLD 전환
5. 정산 READY 복귀

각 액션 카드에 공통 표시:

## 7. 내부 상태 vs PG 상태 비교 UX

### 7.1 비교 패널 3열 구조

추천 비교 영역:

1. 내부 결제 상태
2. 내부 Gateway 원장
3. Toss/PG 실조회 상태

한 줄 요약 문구 예:

- 내부 결제는 완료지만 PG는 취소 상태입니다.
- PG 승인까지 끝났지만 정산 전환이 누락되었습니다.
- 지급 실패로 인해 정산이 보류 상태로 이동해야 합니다.

### 7.2 mismatch 경고 단계

색상 단계 제안:

- 빨강: 즉시 조치 필요
- 주황: 재조회 필요
- 파랑: 배치 대기

### 7.3 mismatch 예시

#### 케이스 A

- 내부 결제 상태: `PAID`
- PG 상태: `DONE`
- 정산 상태: `WAIT`

판단:

- 정산 READY 전환 누락 가능성

추천 버튼:

- `정산 상태 재계산`

#### 케이스 B

- 내부 결제 상태: `CONFIRMED`
- PG 상태: `CANCELED`

판단:

- 내부/외부 상태 충돌

추천 버튼:

- `PG 기준 동기화`
- `분쟁 등록`

#### 케이스 C

- 지급 상태: `FAILED`
- sellerStatus: 비정상
- payoutRef: 존재

판단:

- 판매자 상태 또는 계좌 정보 확인 필요

추천 버튼:

- `지급 재시도`
- `보류 전환`

## 17. 최종 제안

관리자 결제/정산 UI는 단순 리스트가 아니라 운영 판단과 액션 수행이 가능한 콘솔이어야 한다.

핵심은 아래 세 가지다.

1. 상태를 분리해서 보여줄 것
2. 내부/외부 상태 차이를 즉시 드러낼 것
3. 다음 액션과 근거를 같은 화면에서 제공할 것

이 문서 기준으로 구현을 시작하면, 운영자는 다음을 얻을 수 있다.

- 지금 처리해야 할 건이 무엇인지 즉시 파악
- PG/내부 상태 충돌을 빠르게 식별
- 지급 실패와 분쟁을 별도 흐름으로 관리
- 강제 처리와 수동 개입의 근거를 로그로 남김

다음 단계 구현 우선순위는 아래를 권장한다.

1. 통합 리스트 상태 4분리
2. 상세 페이지 비교 패널
3. mismatch 경고 박스
4. 운영 액션 패널
5. 메모/감사 로그
